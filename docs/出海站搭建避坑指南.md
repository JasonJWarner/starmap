## 1. GitHub API 相关

### ❌ 坑 1：GitHub API 未认证限速极严

**严重程度：** 🔴 高

**问题现象：**
```
GET https://api.github.com/search/repositories → 403 Forbidden
"API rate limit exceeded for unauthenticated user"
```

未认证时每小时只能请求 10 次，11 个采集脚本根本无法运行完。

**解决方案：**
申请 GitHub Personal Access Token（见保姆指引 3.3 节），在请求头中加入：
```javascript
headers: {
  'Authorization': 'token ' + process.env.GITHUB_TOKEN
}
```

### ❌ 坑 2：中文关键词搜索效果极差

**严重程度：** 🔴 高

**问题现象：**
用中文关键词搜索 GitHub，返回数量远低于英文关键词。

**根因：** GitHub 搜索索引对中文关键词支持很差，很多项目即使标题是中文也不会被索引到。

**解决方案：**
- 全部使用英文关键词搜索
- 同一个子分类用多个不同关键词覆盖（增加命中概率）
- 采集后人工检查分类准确性

### ❌ 坑 3：GitHub API 请求间隔限制

**严重程度：** 🟡 中

**问题现象：**
即使有 Token，连续请求仍会触发 403：
```
"You have exceeded a secondary rate limit"
```

**解决方案：**
在两个请求之间加 7 秒间隔：
```javascript
await sleep(7000); // 不要小于 7 秒
```

### ❌ 坑 4：per_page 上限为 100

**严重程度：** 🟢 低（已知限制）

**问题现象：**
无论 `per_page` 设多高，GitHub Search API 最多返回 100 条结果。

**解决方案：**
每个关键词最多只能取到 100 条。如果关键词覆盖不足热门领域，需要增加更多细分关键词，或者用 GitHub Ranking 补充。

### ❌ 坑 5：pushed 过滤导致新项目被漏采

**严重程度：** 🟡 中

**问题现象：**
使用 `pushed:>2025-10-19` 过滤后，很多优质老项目（近 6 个月无更新）被排除。

**解决方案：**
- 如果要收录老项目，关键词中不加 `pushed:` 过滤
- 或者维护一份"顶级项目列表"手动补充（不依赖关键词搜索）

## 2. 数据采集相关

### ❌ 坑 6：archived 项目 stars 虚高

**严重程度：** 🔴 高

**问题现象：**
很多 archived 项目 stars 很高（如一些上古教程），采集进来后拉低整体质量。

**解决方案：**
在代码中明确过滤：
```javascript
if (repo.archived) continue;
```

### ❌ 坑 7：fork 数为 0 的项目无意义

**严重程度：** 🔴 高

**问题现象：**
GitHub 上有很多个人仓库创建后从未被 fork，既不是工具也不是学习对象。

**解决方案：**
```javascript
if (repo.forks_count === 0) continue;
```

### ❌ 坑 8：增量保存没做，脚本中断数据全丢

**严重程度：** 🟡 中

**问题现象：**
脚本运行到一半崩溃/被中断，所有数据丢失，必须从头开始。

**解决方案：**
每处理完一个关键词就保存一次：
```javascript
for (const { q, sub } of QUERIES) {
  // ... 处理数据 ...
  // 每轮保存，不等到最后
  fs.writeFileSync(OUT_FILE, JSON.stringify(repos, null, 2), 'utf-8');
  await sleep(7000);
}
```

### ❌ 坑 9：stars 阈值启发式规则不准确

**严重程度：** 🟡 中

**问题现象：**
纯按 stars 数量判断难度有系统误差：
- `awesome-python`（293K stars）→ 被标为"入门"，实际是专业参考手册
- `freeCodeCamp`（443K stars）→ 被标为"入门"，这个反而是对的

**解决方案：**
用关键词修正：
```javascript
function assignDifficulty(repo) {
  const name = repo.n.toLowerCase();

  // awesome 列表 → 专业（内容是高级参考，非入门资料）
  if (name.startsWith('awesome-') && repo.s > 15000) return '专业';

  // stars 阈值
  if (repo.s > 50000) return '专业';
  if (repo.s > 15000) return '进阶';

  // 向量数据库/框架类 → 降档（使用门槛低）
  const desc = (repo.d || '').toLowerCase();
  if (desc.includes('vector database') || name.includes('ollama')) {
    return repo.s > 15000 ? '进阶' : '入门';
  }

  return '入门';
}
```

### ❌ 坑 10：数据合并后出现重复项目

**严重程度：** 🟡 中

**问题现象：**
同一个项目被多个关键词命中，合并后出现重复。

**解决方案：**
```javascript
const seen = new Set();
const unique = allRepos.filter(r => {
  if (seen.has(r.name)) return false;
  seen.add(r.name);
  return true;
});
```

### ❌ 坑 11：子分类 subCategory 为空

**严重程度：** 🟡 中

**问题现象：**
部分 repo 没有匹配到任何关键词，subCategory 为空。

**解决方案：**
检查采集日志，找出哪些关键词没返回数据，调整关键词。

## 3. Git 操作相关

### ❌ 坑 12：SSH 22 端口被封导致 git push 失败

**严重程度：** 🔴 高

**问题现象：**
```
ssh: connect to host github.com port 22: Connection timed out
fatal: Could not read from remote repository
```

国内网络封锁了 GitHub SSH 端口 22。

**解决方案（方案 A - SSH over HTTPS）：**
编辑 `C:\Users\YourName\.ssh\config`：
```
Host github.com
  Hostname github.com
  User git
  Port 443
  ProxyCommand connect -S 127.0.0.1:10808 %h %p
```

**解决方案（方案 B - HTTPS 替代 SSH）：**
```powershell
git remote set-url origin https://github.com/Username/repo.git
```
然后每次 push 时输入用户名和 Personal Access Token。

### ❌ 坑 13：Vercel webhook 漏推，push 后不自动部署

**严重程度：** 🔴 高

**问题现象：**
`git push` 成功，但 Vercel 没有触发新部署，线上版本落后几个 commit。

**解决方案：**
用 Vercel CLI 手动触发：
```powershell
vercel --token YOUR_TOKEN --prod --yes
```

**预防措施：**
在 Vercel Dashboard → Project → Settings → Git → 检查 webhook 是否正确连接。

### ❌ 坑 14：PowerShell 中 `&&` 链式命令不支持

**严重程度：** 🟡 中

**问题现象：**
```powershell
node script1.js && node script2.js  # ❌ 报错
```

**解决方案：**
```powershell
node script1.js; node script2.js    # ✅ 用分号
```

或者用 PowerShell 的 `&`：
```powershell
& node script1.js; & node script2.js
```

### ❌ 坑 15：PowerShell 中 `head` 命令不存在

**严重程度：** 🟡 中

**问题现象：**
```powershell
curl ... | head -n 10  # ❌ head 命令不存在
```

**解决方案：**
```powershell
curl ... | Select-Object -First 10
```

### ❌ 坑 16：PowerShell 中 `-H` 参数语法不兼容 curl

**严重程度：** 🟡 中

**问题现象：**
```powershell
curl -H "Authorization: Bearer xxx" ...  # ❌ PowerShell curl 别名
```

**解决方案：**
```powershell
curl.exe -H "Authorization: Bearer xxx" ...  # 用 curl.exe 绕过别名
# 或
Invoke-RestMethod -Headers @{...} ...
```

### ❌ 坑 17：PowerShell 编码导致 git add 失败

**严重程度：** 🟡 中

**问题现象：**
PowerShell 默认输出编码为 GBK，与 git 期望的 UTF-8 不兼容，导致特殊字符处理失败。

**解决方案：**
```powershell
$OutputEncoding = [Console]::OutputEncoding = [System.Text.Encoding]::UTF8
[Console]::InputEncoding = [System.Text.Encoding]::UTF8
git add -A
```

## 4. Vercel 部署相关

### ❌ 坑 18：Vercel Token 过期导致 CLI 无法使用

**严重程度：** 🔴 高

**问题现象：**
```
Error: The specified token is not valid. Use `vercel login` to generate a new token.
```

**解决方案：**
Vercel Dashboard → Settings → Tokens → 创建新 Token，然后：
```powershell
vercel --token NEW_TOKEN --prod --yes
```

### ❌ 坑 19：`vercel build` 失败 — prebuilt 输出不存在

**严重程度：** 🟡 中

**问题现象：**
```
Error: The "--prebuilt" option was used, but no prebuilt output found in ".vercel/output"
```

**解决方案：**
对于纯静态网站，Vercel 会自动推断构建命令，不需要 `--prebuilt`：
```powershell
vercel --token YOUR_TOKEN --yes  # 不加 --prebuilt
```

### ❌ 坑 20：Vercel 自动部署失败后无通知

**严重程度：** 🟡 中

**问题现象：**
push 了代码，Vercel 也在构建，但用户不知道构建是否成功。

**解决方案：**
在 Vercel Dashboard → Project → Settings → Notifications 中配置：
- 开启 "Deployment Failed" 邮件通知
- 或接入 Slack/Discord 通知

### ❌ 坑 21：Vercel hobby 计划只能用 Public 仓库

**严重程度：** 🟢 低（已知限制）

**问题现象：**
Vercel 免费版不支持 Private 仓库自动部署。

**解决方案：**
- 使用 Public 仓库（代码开源）
- 或付费升级到 Pro 计划（$20/月）

## 5. 前端开发相关

### ❌ 坑 22：Hash 路由对 SEO 不友好

**严重程度：** 🔴 高

**问题现象：**
`https://example.com/#/category/ai` 这种 URL，Google 爬虫不会正确解析后面的 hash 部分。

**解决方案：**
- 如果 SEO 非常重要，改为 History API 路由（需要服务器配置 fallback）
- 接受 Hash 路由限制，在 meta 标签和 sitemap 中优化可索引内容
- Google Search Console 可以主动提交 hash URL，不需要爬虫发现

### ❌ 坑 23：localStorage 在隐私模式下不可用

**严重程度：** 🟡 中

**问题现象：**
iOS Safari 的隐私模式或其他浏览器的无痕模式下，`localStorage.setItem()` 会抛出异常。

**解决方案：**
```javascript
try {
  localStorage.setItem('key', 'value');
} catch (e) {
  // 静默失败，不影响主流程
}
```

### ❌ 坑 24：app.js edit 操作因 whitespace 不匹配失败

**严重程度：** 🟡 中

**问题现象：**
用 edit 工具修改 app.js 时，`oldText` 精确匹配失败（因为 whitespace 差异）。

**解决方案：**
- 大改动用 write 工具整体重写文件
- 或者用 exec 调用 sed/PowerShell 做精确替换

### ❌ 坑 25：JSON 字段名过长导致 catalog.json 体积大

**严重程度：** 🟢 低（已知优化点）

**问题现象：**
每个 repo 30+ 字段，catalog.json 超过 1.5MB，影响加载速度。

**解决方案：**
精简字段名（`name` → `n`, `description` → `d`, `stars` → `s`），见 `prepare-site-data.js`。

## 6. SEO 相关

### ❌ 坑 26：`lang="zh-CN"` 严重影响国际化搜索

**严重程度：** 🔴 高

**问题现象：**
中文网站在 Google 英文搜索结果中排名极差，即使内容优质。

**解决方案：**
```html
<html lang="en" dir="ltr">
<!-- 同时声明多语言版本 -->
<link rel="alternate" hreflang="en" href="https://yoursite.com/">
<link rel="alternate" hreflang="zh-CN" href="https://yoursite.com/?lang=zh">
<link rel="alternate" hreflang="x-default" href="https://yoursite.com/">
```

### ❌ 坑 27：中文 sitemap 影响 Google 抓取

**严重程度：** 🟡 中

**问题现象：**
sitemap 中的 URL 如果包含中文，Google 可能无法正确解析。

**解决方案：**
- sitemap 中用 URL 编码的路径，或直接用英文 slug
- 分类页 URL 用英文标识符：`/#/category/ai-tools-models` 而非 `/#/category/AI工具`

### ❌ 坑 28：上线前未提交 Google Search Console

**严重程度：** 🟡 中

**问题现象：**
网站上线后 0 访问，Google 从未发现过该站点。

**解决方案：**
1. 注册 Google Search Console
2. 验证域名所有权（Vercel 域名通常自动验证）
3. 提交 sitemap.xml
4. 等待 1~7 天 Google 抓取

### ❌ 坑 29：meta description 为中文，英文搜索不展示

**严重程度：** 🟡 中

**问题现象：**
Google 搜索结果摘要显示乱码或空白。

**解决方案：**
所有 meta 标签提供英文版本（见保姆指引第 7 节 SEO 配置）。

### ❌ 坑 30：没注册 Bing Webmaster

**严重程度：** 🟢 低（容易被忽略）

**问题现象：**
只关注 Google，忽略了 Bing（覆盖约 3%~5% 的搜索流量）。

**解决方案：**
同步注册 Bing Webmaster Tools：https://www.bing.com/webmasters

## 7. 数据同步相关

### ❌ 坑 31：master-sync.js 并行执行多个实例

**严重程度：** 🔴 高

**问题现象：**
两个同步进程同时运行，data 目录下文件被互相覆盖，catalog 数据损坏。

**根因：**
cron 触发一次，手动又触发一次，两个进程同时读写同一批文件。

**解决方案：**
在 master-sync.js 开头加锁文件检查：
```javascript
const LOCK_FILE = path.join(__dirname, '..', 'logs', 'sync.lock');
if (fs.existsSync(LOCK_FILE)) {
  console.log('Sync already running, exiting.');
  process.exit(0);
}
fs.writeFileSync(LOCK_FILE, process.pid.toString(), 'utf-8');
// ... 执行完成后 ...
fs.unlinkSync(LOCK_FILE);
```

### ❌ 坑 32：catalog.json 和 site/data/catalog.json 不同步

**严重程度：** 🟡 中

**问题现象：**
采集脚本更新了 `data/` 目录，但站点加载的是 `site/data/catalog.json`，数据过期。

**解决方案：**
确保 master-sync.js 最后一步执行 `prepare-site-data.js`，将数据从 `data/` 同步到 `site/data/`。

### ❌ 坑 33：tags.json 与 catalog.json 的 key 不匹配

**严重程度：** 🟡 中

**问题现象：**
tags.json 有 2319 条记录，但实际只有 87% 的 repo 能匹配到标签。

**根因：**
tags.json 的 key 格式（如 `owner/repo`）与 catalog.json 中 repo.n 的格式不一致。

**解决方案：**
写一个 `fill-missing-tags.js` 脚本，专门检查两边 key 是否完全匹配，补全缺失项。

### ❌ 坑 34：Umami 没有访问记录

**严重程度：** 🟡 中

**问题现象：**
网站已上线，Umami Dashboard 显示 0 访问。

**排查方向：**
1. Umami Cloud 账号是否正确登录了正确的 Website
2. index.html 中 script 标签是否正确（Website ID 是否匹配）
3. 是否在浏览器中打开了 DevTools 检查 Network 请求（确认 umami.is 请求发出）
4. 是否是浏览器 AdBlock 拦截了 umami.is

## 8. 网络与代理相关

### ❌ 坑 35：系统代理影响 GitHub API 调用

**严重程度：** 🔴 高

**问题现象：**
`curl https://api.github.com/...` 超时，但浏览器能访问 GitHub。

**根因：**
系统设置了 HTTP 代理，但 Node.js 脚本默认不使用系统代理。

**解决方案：**
```javascript
// 在 Node.js 中设置代理
process.env.HTTPS_PROXY = 'http://127.0.0.1:10808';
process.env.HTTP_PROXY = 'http://127.0.0.1:10808';
```

### ❌ 坑 36：Git 配置了代理但 SSH 仍走直连

**严重程度：** 🟡 中

**问题现象：**
设置了 `git config --global http.proxy`，但 `git push` 仍然超时。

**解决方案：**
SSH 不走 HTTP 代理，需要单独配置 SSH 走代理（见坑 12）。

### ❌ 坑 37：npm install 安装失败

**严重程度：** 🟡 中

**问题现象：**
```powershell
npm install  # ERR: network timeout or 404
```

**解决方案：**
配置 npm 镜像：
```powershell
npm config set registry https://registry.npmmirror.com
```
